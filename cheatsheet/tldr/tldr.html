DevOps (Overview)

cultureautomationmetrics

TL;DR: DevOps is a way of working that tightens the loop between building and running software—optimize the system, not the silos.

Read the deep dive

DevOps blends culture, practices, and tooling to increase flow (from idea → value), improve feedback, and build continual learning. Core practices include trunk-based development, CI/CD, automated testing, infrastructure as code, observability, and blameless postmortems.

  • Outcomes: shorter lead time, higher deployment frequency, lower change-failure rate, faster MTTR.
  • Start small: one product team, value stream mapping, remove the top 1–2 bottlenecks, measure, repeat.

Infrastructure as Code (IaC)

repeatabilityidempotentreviewable

TL;DR: Treat infrastructure like application code—versioned, reviewed, tested, and promoted through environments via pipelines.

Read the deep dive

IaC improves consistency and speed by describing infra declaratively (what) or imperatively (how). Store it in Git, run through CI/CD, and validate with tests and policy-as-code before applying changes.

  • Patterns: immutable infra, drift detection, environment parity, modules for reuse, and policy guardrails.
  • Anti-patterns: manual hotfixes in portals, snowflake servers, copying code between repos instead of modules.

Terraform

HCLmodulesstate

TL;DR: A declarative IaC tool using HCL; plan changes, review in PRs, then apply via pipelines with remote state and policies.

Read the deep dive

Use terraform plan to preview changes and apply to execute. Keep state remote (e.g., Azure Storage + lock with Azure Table/Blob or Terraform Cloud). Encapsulate patterns with versioned modules.

# Minimal Azure Resource Group
terraform {
  required_providers { azurerm = { source = "hashicorp/azurerm", version = ">= 3.0" } }
}
provider "azurerm" { features {} }
resource "azurerm_resource_group" "rg" {
  name     = "rg-demo"
  location = "westeurope"
}
  • Validate with terraform validate + tflint; secure with policy (OPA/Conftest, Sentinel, or Azure Policy).
  • Structure repos with environments and modules; avoid copy-paste by publishing modules.

Azure Bicep

AzureARMmodules

TL;DR: A concise DSL for Azure resources that compiles to ARM—great for Azure-first teams with native tooling.

Read the deep dive

Author resources with type-safe IntelliSense, reuse with modules, and preview changes using what-if before deploy.

// storage.bicep
resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'stgdemotldr'
  location: 'westeurope'
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
}
  • Best with Azure DevOps or GitHub Actions; enforce with Azure Policy and RBAC scoping.

CI/CD Pipelines

testingrunnersgates

TL;DR: Automate build, test, security scans, and promotion. Every change flows through the same gates.

Read the deep dive

Prefer trunk-based development with short-lived branches. Use reusable pipeline templates and environment-specific variables and identities.

# GitLab CI example for Terraform
stages: [fmt, validate, plan, apply]

fmt:
  stage: fmt
  script: terraform fmt -check -recursive

validate:
  stage: validate
  script: |
    terraform init -backend-config=backend.hcl
    terraform validate

plan:
  stage: plan
  script: terraform plan -out tf.plan
  artifacts:
    paths: [tf.plan]

apply:
  stage: apply
  when: manual
  script: terraform apply -auto-approve tf.plan

Secrets & Policy

VaultKey VaultOPA

TL;DR: Keep secrets out of repos; inject just‑in‑time via identity. Enforce platform guardrails with policy‑as‑code.

Read the deep dive
  • Secrets: use cloud KMS (Azure Key Vault) or Vault; bind via OIDC/workload identity, never long‑lived keys.
  • Policy: validate IaC with OPA/Conftest or native policies; stop drift and non‑compliant resources.

Observability & Telemetry

metricslogstraces

TL;DR: Measure what matters. Use metrics, logs, and traces to understand behaviors and catch issues early with actionable alerts and SLOs.

Read the deep dive
  • Three pillars: metrics (quantitative), logs (context), traces (causality across services).
  • Outcomes: faster MTTR, fewer pagers, better capacity planning.
  • Practices: RED/USE methods, meaningful alerting, runbooks, SLOs & error budgets.

Site Reliability Engineering (SRE)

SLOserror budgetstoil

TL;DR: Engineer reliability with SLOs and error budgets; automate repetitive toil and learn from incidents via blameless postmortems.

Read the deep dive
  • Plan: pick user-centric SLI/SLOs; track error-budget burn.
  • Operate: lightweight change management, progressive delivery, runbooks.
  • Improve: incident reviews → actions → reduce repeat failures.

Continuous Testing

shift-leftautomationquality

TL;DR: Test early and often: unit → integration → e2e. Automate performance, security, and contract tests in CI.

Read the deep dive
  • Pyramid: many fast unit tests; fewer, broader e2e. Keep flake low.
  • Gates: coverage thresholds, SAST/DAST, SBOM, dependency checks.
  • Data: ephemeral envs & seeded test data; mock external services.

Release Strategies

canaryblue‑greenfeature flags

TL;DR: Reduce risk with progressive delivery—route a slice of traffic first; promote on healthy signals; instant rollback on regressions.

Read the deep dive
  • Patterns: blue‑green, canary, rings, dark launches, A/B.
  • Signals: error rate, latency, business KPIs, user feedback.
  • Flags: decouple deploy from release; kill switches.

Value Stream Mapping

flowlead timebottlenecks

TL;DR: Visualize the path from idea → production; expose wait states and rework; fix the biggest constraint first.

Read the deep dive
  • Track lead time, deployment frequency, change fail %, MTTR.
  • Make work visible; limit WIP; measure again after changes.

Platform Engineering

IDPgolden pathsDX

TL;DR: Provide self‑service, secure paved roads (golden paths) so teams ship fast without reinventing the wheel.

Read the deep dive

  • Capabilities: templates, pipelines, environments, guardrails, docs.
  • Outcome: reduced cognitive load, higher flow efficiency.

Chaos Engineering

resiliencegame daysrecovery

TL;DR: Intentionally inject failure in controlled ways to validate resilience and improve recovery procedures.

Read the deep dive
  • Start with a hypothesis tied to an SLO; scope small; automate experiments.
  • Practice incident response via game days; capture learnings and fixes.
